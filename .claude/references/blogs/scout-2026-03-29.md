### Proposal 1 — Simplify allow-claude-edits hook with conditional `if` field

**File:** `.claude/hooks/allow-claude-edits.js` + `.claude/settings.json`
**What:** Replace manual tool_name + file_path checking with v2.1.85's `if` field
**Why:** v2.1.85 added conditional `if` field for hooks using permission rule syntax. Our hook manually parses stdin JSON to check tool name and path — the `if` field can do this declaratively.
**Diff:**
```
old (settings.json hook config):
  hook entry runs allow-claude-edits.js unconditionally, script checks tool+path

new:
  hook entry uses "if": "Edit(.claude/skills/**) || Write(.claude/skills/**) || Edit(.claude/specs/**) || ..."
  to filter before the script runs — script only needs to emit the permissionDecision
```

### Proposal 4 — Document new hook events in /create-hook

**File:** `.claude/skills/create-hook/SKILL.md`
**What:** Add 6 new hook events to the reference list
**Why:** v2.1.69–v2.1.85 added `TaskCreated`, `CwdChanged`, `FileChanged`, `StopFailure`, `PostCompact`, `InstructionsLoaded` — users creating hooks need to know these exist.
**Diff:**
```
old: Hook event list ends at existing events

new: Add to event reference:
- TaskCreated (v2.1.84) — fires when a task is created; exit 2 to prevent
- CwdChanged (v2.1.83) — fires when working directory changes
- FileChanged (v2.1.83) — fires when a file is modified
- StopFailure (v2.1.78) — fires when turn ends from API error
- PostCompact (v2.1.76) — fires after conversation compaction
- InstructionsLoaded (v2.1.69) — fires when CLAUDE.md/rules are loaded
- Elicitation / ElicitationResult (v2.1.76) — MCP elicitation hooks
```
